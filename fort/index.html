The gateway model

Issue tokens in Fort. Verify them everywhere.

Your products never hold the JWT secret. They send the bearer token to one endpoint — Fort answers with verified claims.

One seam, no secret sprawl

Verification is a single HTTP call

Every downstream service — in any language — validates a bearer token with one request. No shared secret in a dozen env files, no key distribution, no re-implemented JWT parsing. Rotate the secret in one place and every consumer keeps working.

# Any product, any language
POST https://fort.example.com/v1/token/verify

Multi-platform white-label

One deployment, many products

Fort serves any number of consuming apps — "platforms" — from a single deployment. Each gets its own branding, enabled login methods, settings, allowed-user lists and API tokens, with a hardened tenant resolver enforcing isolation on every query.

Supabase-compatible

Same claims, no Supabase project required

Fort issues HS256 JWTs with Supabase claims and verifies GoTrue ES256 tokens, so an existing Supabase ecosystem keeps working. Migrate off managed Supabase Auth by changing only the issuer and secret — zero consumer-side payload changes.

For developers

Ship it with real SDKs

Fort's SDKs are the same ones its own platform products run on: @sentra/fort-auth-sdk (TypeScript + Go) with middleware, role guards and ready-made auth & lock-screen widgets — plus admin, billing and storage SDK families, and a TypeScript umbrella package.

Open the developer portal

// Go — protect a route with Fort
cfg := fort.Config{BaseURL: "https://fort.example.com"}

mux.Handle("/admin",
  fort.Middleware(cfg)(
    fort.RequireRole("admin")(adminHandler)))

claims, _ := fort.ClaimsFromContext(r.Context())
