SDKs

Fort ships a real, working SDK family — four packages split by concern, plus an umbrella that re-exports them all. TypeScript packages target Next.js/React products (with embeddable UI widgets); Go packages give backend services middleware and typed clients.

The family at a glance

PackageTypeScriptGoWhat it covers
@sentra/fort-auth-sdkYes — clients, hooks, widgetsYes (package fort)Auth core: login/session clients, token verification, middleware, branding, profile, auth + lock-screen widgets.
@sentra/fort-admin-sdkYes — clients + admin pagesYes (package fortadmin)Admin users, roles, settings and feature-flags clients; drop-in admin UI pages; role-check helpers.
@sentra/fort-billing-sdkYesYes (package fortbilling)Usage metering, invoices, price-books, customers.
@sentra/fort-storage-sdkYes—Media/object client + media-library widget. TypeScript only.
@sentra/fort-sdkYes (umbrella)—Re-exports all four TS families as one dependency.
Go note. There is no single top-level Go module — each family ships its own Go package under its directory (sdk/auth/go = fort, sdk/admin/go = fortadmin, sdk/billing/go = fortbilling). Storage has no Go client.

Install

Each TS package requires Node 18+ and comes with cross-platform install scripts that build the package (and its sibling dependencies) and verify dist/:

# macOS / Linux (or Git Bash) — from the package directory
./install.sh                 # build in place, ready to import
./install.sh /path/to/app    # build + install into that app

# or as a git dependency
npm install github:sentra-intel/fort-auth-sdk

@sentra/fort-auth-sdk TS Go

TypeScript exports: FortAuthClient, FortBrandingClient/fetchBranding, FortProfileClient, verifyToken/verifyHS256, fortMiddleware (Next.js middleware), the FortProvider context (useFort, useFortAuth, useFortMeta), and UI widgets: FortAuthWidget, FortUserMenu, FortProfileWidget, FortProfilePages, FortLockScreen, FortPasswordLockScreen.

// Next.js — wrap the app, drop in the login widget, gate routes
import { FortProvider, FortAuthWidget, fortMiddleware } from '@sentra/fort-auth-sdk'

<FortProvider baseUrl={process.env.NEXT_PUBLIC_FORT_URL}>{children}</FortProvider>
<FortAuthWidget onAuthenticated={() => location.assign('/dashboard')} />

// middleware.ts — verify the session cookie on every request
export const middleware = fortMiddleware({ baseUrl: process.env.FORT_URL })

The provider fetches /v1/branding before login, so the widget renders your platform's name, colors and enabled login methods with zero configuration — OTP and 2FA steps included.

Go package (fort)

Exports: NewClient/Config (BaseURL, Timeout, JWTSecret, PlatformToken), Client.VerifyToken, VerifyHS256, ExtractBearer, Middleware, RequireRole, ClaimsFromContext, and Claims with HasRole.

// Guard your service's routes with Fort — HTTP verify, no shared secret
cfg := fort.Config{BaseURL: "https://fort.example.com"}

mux.Handle("/admin",
    fort.Middleware(cfg)(
        fort.RequireRole("admin")(adminHandler),
    ),
)

// Read the verified identity downstream
claims, ok := fort.ClaimsFromContext(r.Context())
if ok {
    userID := claims.Sub                      // never trust a client-supplied id
    isOps  := claims.HasRole("security_operator")  // checks Role + Roles[]
}
Two verification modes, one switch. Leave Config.JWTSecret empty and Middleware verifies over POST /v1/token/verify (recommended). Set it, and verification happens locally with VerifyHS256 — for deployments that already share the secret. Set Config.PlatformToken (from FORT_PLATFORM_TOKEN) and every call carries your X-Fort-Platform identity.
RequireRole checks the primary role claim only. When your access rule should also match RBAC-granted roles in roles[], check claims.HasRole(…) in your handler (or use the admin package's RequireAnyRole).

@sentra/fort-admin-sdk TS Go

TypeScript: FortAdminUsersClient, FortRolesClient, FortAdminSettingsClient, FortFlagsClient, plus full admin UI pages — FortAdminUsersPages, FortRolesPages, FortAdminSettingsPages — so a consuming product gets a user-management screen without building one.

import { FortAdminUsersClient } from '@sentra/fort-admin-sdk'

const users = new FortAdminUsersClient({ baseUrl: '/api/v1', platform: 'acme' })
await users.list({ limit: 50 })

Go (fortadmin): FlagsClient (Enabled, Value — cached feature-flag reads), SettingsClient.Get, RequireAnyRole(…) middleware, and claim helpers ClaimsIsAdmin / ClaimsIsSuperAdmin.

@sentra/fort-billing-sdk TS Go

Usage metering and invoicing against Fort's billing subsystem, callable from any product:

// TypeScript
import { FortBillingClient } from '@sentra/fort-billing-sdk'

const billing = new FortBillingClient({
  baseUrl: process.env.FORT_URL,
  platformToken: process.env.FORT_PLATFORM_TOKEN,
})
await billing.recordUsage({ metric: 'tokens', quantity: 1280, source: 'reporting' })
const { items: invoices } = await billing.listInvoices('acme')

Go (fortbilling): NewClient(fort.Config) with RecordUsage(ctx, UsageEvent) and GenerateInvoice(ctx, accessToken, GenerateInvoiceInput).

@sentra/fort-storage-sdk TS

Media/object client and a ready-made media-library widget (TypeScript only):

import { FortMediaClient, FortMediaLibrary } from '@sentra/fort-storage-sdk'

const media = new FortMediaClient({ baseUrl: '/api/v1' })
const asset = await media.upload(file)
<FortMediaLibrary onSelect={(a) => console.log(a.url)} />

@sentra/fort-sdk — the umbrella TS

One dependency that re-exports the auth, admin, storage and billing families. Use it when a product consumes several families; depend on individual packages when you want a smaller surface. Peer dependencies (next, react, and the @prism/ui design system for widgets) are optional — server-side usage works without them.
