Events & NATS

Fort runs its own NATS server and publishes auth lifecycle events on fort.* subjects. Consuming products subscribe to react the moment auth state changes — provision users, tear down cached sessions, refresh branding — instead of polling the REST API.

Why events, not another database

Fort's product boundary is strict: other products never read fort_db. Everything crosses one of two public seams — token verification over HTTP, and auth events over NATS. That keeps every consumer decoupled: you can rebuild your user cache from events without ever coupling to Fort's schema.

Subjects

Payloads are plain JSON, verified against the publisher in Fort's internal/natsseam package:

SubjectPublished whenPayload
fort.user.created A new user is seeded or provisioned {"user_id":"<uuid>","email":"<email>"}
fort.session.revoked All refresh tokens for a user are revoked (password reset, admin revoke, logout-everywhere) {"user_id":"<uuid>","revoked_count":N,"reason":"<reason>"}
fort.branding.updated An admin patches branding (PATCH /v1/admin/branding) {"updated_by":"<uuid>"}
fort.platform.member.added A user is added to a platform's member list {"user_id":"<uuid>","email":"<email>","platform_id":"<uuid>","role":"<role>"}

Subscribing from a consuming product

Point a NATS client at Fort's NATS URL (FORT_NATS_URL — nats://localhost:4245 in local dev) and subscribe to the subjects you care about:

// Go — react to session revocations and new platform members
nc, err := nats.Connect(os.Getenv("FORT_NATS_URL"))
if err != nil { log.Fatal(err) }

nc.Subscribe("fort.session.revoked", func(m *nats.Msg) {
    var ev struct {
        UserID       string `json:"user_id"`
        RevokedCount int64  `json:"revoked_count"`
        Reason       string `json:"reason"`
    }
    if json.Unmarshal(m.Data, &ev) == nil {
        dropCachedSessions(ev.UserID)   // your side: evict, disconnect, audit
    }
})

nc.Subscribe("fort.platform.member.added", func(m *nats.Msg) {
    // eagerly provision the user in your product so they're visible
    // before their first sign-in
})
A real pattern: subscribing to fort.platform.member.added lets a product create its own user record the moment an admin grants access in Fort — the user's workspace exists before they ever log in.

Events vs. polling

NeedUse
Know a session was revoked (kill live connections, evict caches)Subscribe to fort.session.revoked
Provision users into your product as they're addedSubscribe to fort.user.created / fort.platform.member.added
Refresh cached branding when an admin re-themesSubscribe to fort.branding.updated, then re-fetch GET /v1/branding
Authorize an individual request right nowPOST /v1/token/verify — always the API, never events
Reconcile full state (startup, recovery, audits)The REST API — events are a signal, not a system of record

Delivery semantics

Events are best-effort, fire-and-forget. Fort publishes with plain NATS publish calls — non-blocking, and silently skipped if NATS is unavailable — so the auth flow never stalls on the event stream. Treat events as a fast-path optimization: keep a reconcile path against the REST API for anything correctness-critical, and never rely on an event you might not receive to enforce security. (Token verification, not revocation events, is the authoritative check.)
Inbound webhooks: Fort's event seam is NATS-native — there is no signed-HTTP webhook delivery today. If your consumer can't speak NATS, run a small bridge process that subscribes and forwards to your own HTTP endpoint. The repository also contains a webhook dispatcher module (internal/auth/webhook_dispatcher.go); see the repository docs for its current status before building on it.

Local development

# make start brings NATS up alongside fort_db (JetStream-enabled, :4245)
make start

# tail Fort events with the NATS CLI
