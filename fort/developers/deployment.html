Self-hosting

Fort is built to be run by you: one Go binary, one PostgreSQL database, one NATS server, and an optional Next.js dashboard. No SaaS dependency, no per-seat vendor bill — your auth authority runs where your data lives.

One binary, five subcommands

fort serve      # production: HTTP gateway on FORT_HTTP_ADDR
fort start      # dev: docker compose up (db + NATS) → migrate → serve
fort migrate    # apply Atlas migrations (incremental, idempotent)
fort reset      # drop + re-apply all migrations
fort seed       # seed admin user + platform rows + fixtures

# plus a plugin CLI for DB-driven plugins:
fort plugin list | enable | disable | generate

The repository's Makefile wraps these: make start, make migrate, make seed, make test, make build (→ ./bin/fort).

Ports

Default local topology (all four are configurable through the env contract below):

PortComponentNotes
:8213Fort HTTP gatewayThe API surface — REST, gateway core, MCP servers.
:5445fort_db — ParadeDB (Postgres 17)Fort's own database. No other product connects to it — ever.
:4245Fort NATS (JetStream)Publishes fort.* events.
:3213Web dashboard (Next.js)Optional admin UI; proxies /api/v1/* to the gateway — the browser never calls :8213 directly.

Environment contract

VariableRequiredWhat it does
FORT_HTTP_ADDRYesGateway listen address (default :8213).
FORT_DATABASE_URLYesPostgres DSN for fort_db (ParadeDB pg17; localhost:5445 locally).
FORT_NATS_URLYesFort's NATS server (nats://localhost:4245; wss:// also supported).
FORT_JWT_SECRETYesHS256 signing secret. The single most sensitive value in the deployment — see the checklist below.
FORT_CORS_ORIGINSYesComma-separated browser-origin allowlist. Never *.
TOTP_ENCRYPTION_KEYFor 2FAEncrypts stored TOTP secrets; required before any 2FA enroll/verify works. Generate with openssl rand -base64 32.
AUTH_EMAIL_DRIVERFor OTP / forgotEmail delivery driver (e.g. smtp; log for dev). OTP, magic-link and forgot-password endpoints depend on it.
APP_ENVYesSet production to disable the dev conveniences (dev admin seed, one-click dev-login).
NEXT_PUBLIC_FORT_URLDashboardWhere the web dashboard's rewrite sends /api/v1/*.

Optional seams (each degrades gracefully when unset): FORT_GOTRUE_URL (self-hosted GoTrue interop), MOTOR_INGEST_URL/MOTOR_DSN (log shipping/observability), CORTEX_HTTP_ADDR/CORTEX_TOKEN (AI seam), FORT_STORAGE_URL/FORT_STORAGE_KEY (storage backend). A commented template ships as .env.example in the repository.

Database & migrations

db/schema.sql is the canonical declared schema; Atlas (db/atlas.hcl) diffs it into ordered migration files under db/migrations/, applied by fort migrate. Query code for the newer domains is generated with sqlc from db/queries/. Fort bootstraps the Supabase-convention auth.users table itself — it does not run a GoTrue daemon by default, but a self-hosted GoTrue can own the same schema and Fort's handlers work against either layout.

make migrate     # apply pending migrations
make atlas       # diff schema.sql → a new migration file
make sqlc        # regenerate query code

Health & first run

GET /health — {"status":"ok","service":"fort"}; unauthenticated, suitable for liveness/readiness probes and uptime checks

On a fresh production deployment there is no dev seed. Bootstrap the first admin through the self-gating setup endpoints, then sign in normally:

curl https://fort.example.com/v1/setup/state
# {"is_first_run":true}

curl -X POST https://fort.example.com/v1/setup/admin \
  -H 'Content-Type: application/json' \
  -d '{"email":"admin@yourco.com","password":"…12+ chars, upper/lower/digit/symbol…"}'
# {"created":true,"user_id":"…"} — 403 setup_already_complete from then on

Production checklist

Generate a strong FORT_JWT_SECRET and treat it as radioactive. Anyone holding it can mint valid sessions. Keep it in a secret manager, never in images or repos — and remember the whole point of the HTTP verify seam is that no other service needs it.
Set APP_ENV=production. This disables the dev admin seed and the one-click dev-login endpoint. Verify neither is reachable before going live.
Set the feature-gate vars before users need them. 2FA enrollment fails without TOTP_ENCRYPTION_KEY; OTP and forgot-password fail without AUTH_EMAIL_DRIVER. Configure both at deploy time, not after the first support ticket.
Lock down the edges. FORT_CORS_ORIGINS = exactly your app origins. fort_db reachable only by Fort. Rate-limit /v1/token/verify at your proxy layer. Rotate platform tokens when staff or vendors change.
Wire your probes and events. Point uptime monitoring at GET /health, and have consuming products subscribe to fort.session.revoked so revocations propagate immediately.
