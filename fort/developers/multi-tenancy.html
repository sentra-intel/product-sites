Platforms & white-label

One Fort deployment serves many consuming apps — Fort calls them platforms. Each platform gets its own branding, settings, allowed users, API tokens and role assignments, isolated at the query layer. That's how one auth authority powers a whole product suite, or a white-label SaaS with a tenant per customer.

How a request resolves to a platform

Every incoming request is resolved to exactly one platform before any per-tenant data is touched. The contract has three steps, in order:

StepSignalHow it matches
1. TokenX-Fort-Platform header or ?platform= query paramThe platform API token is SHA-256-hashed and looked up against enabled platforms. Strongest signal — use it for service-to-service calls.
2. OriginOrigin header, or the Referer originMatched against each platform's registered allowed_origins and domains. This is how browser traffic from your app's domain lands on the right tenant automatically.
3. Default—Falls back to the platform with slug default. If none exists, that's a server configuration error (500) — never a silent cross-tenant read.
The default platform is not token-reachable — by design. Token resolution explicitly excludes slug='default', so a leaked or guessed placeholder token can never resolve to the default tenant. A real tenant needs its own slug and a randomly minted token.

The isolation guarantee

The resolved platform id is a mandatory predicate: every per-platform query inside Fort carries WHERE platform_id = $resolved. A request resolved for platform A cannot read platform B's branding, settings, users or tokens — cross-platform reads are treated as a security defect, not a feature gap. Role resolution is per-platform too: the same user can be admin in one platform and viewer in another, and the access token is minted with the role, roles[] and platform slug for the platform being logged into.

Per-platform branding

Branding is public (pre-login) so login screens can be themed before any session exists:

GET /v1/branding — resolved platform's branding; the SDK passes ?platform=<slug> explicitly
PATCH /v1/admin/branding — partial update (validated hex colors); publishes fort.branding.updated
curl 'https://fort.example.com/v1/branding?platform=acme'
# name, colors, icon and login-method visibility for the "acme" platform —
# the <FortProvider> consumes this to render a fully branded login widget

Admins can also target a specific platform when updating: PATCH /v1/admin/branding?platform=<id>; otherwise the request-resolved platform is used.

Per-platform settings & access control

Settings live in four namespaces — auth_policy (lockout thresholds), session, providers (which login methods are on), system — managed over /v1/admin/settings. Access control adds two allowlists per platform:

  • Allowed users (/v1/admin/allowed-users) — explicit email allowlist for closed deployments.
  • Domain allowlist (/v1/admin/domains) — which email domains may register; the SDK's registration flow fail-closes against GET /v1/auth/domain-check before even attempting signup.

Managing platforms & API tokens

GET /v1/admin/platforms — list; token hashes are never returned
POST /v1/admin/platforms — create; the platform API token is returned once
PATCH /v1/admin/platforms/{id} — update name, origins, domains, enabled
DELETE /v1/admin/platforms/{id} — delete (blocked for default)
POST /v1/admin/platforms/{id}/rotate-token — mint a replacement token (returned once)
GET /v1/me/platforms — the authenticated user's platforms (powers a platform switcher)

Your consuming service stores its token as FORT_PLATFORM_TOKEN and sends it on every Fort call — the Go SDK does this automatically via Config.PlatformToken, the TS admin/billing clients via their platform/platformToken options.

Platform tokens are credentials. Fort stores only their SHA-256 hash and shows the plaintext exactly once at create/rotate time. Keep them server-side, rotate on any suspicion of exposure, and never embed them in browser bundles — browser traffic should resolve by Origin instead.

Wiring a new white-label tenant

  1. Create the platform — POST /v1/admin/platforms with slug, allowed origins and domains. Store the returned token as the tenant service's FORT_PLATFORM_TOKEN.
  2. Brand it — PATCH /v1/admin/branding?platform=… with the tenant's name, colors and icon.
  3. Set its policy — enabled login methods in the providers namespace; lockout policy in auth_policy; domain allowlist if registration is restricted.
  4. Point the app at Fort — the SDK's <FortProvider> fetches /v1/branding?platform=<slug> and renders the tenant's login experience; the backend verifies tokens with the platform token attached.
  5. Add members — platform membership additions emit fort.platform.member.added, so the tenant app can provision users eagerly.
No new deployment per tenant. Adding a platform is a set of API calls against the running Fort — no rebuild, no separate database, no second auth stack to operate.
