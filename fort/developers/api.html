REST API reference

Every endpoint Fort serves, grouped as in the OpenAPI 3.1 spec (api/openapi.yaml in the repository). Examples use https://fort.example.com — replace with your deployment. Cookie-authenticated mutations require an X-CSRF-Token header; Bearer callers are exempt.

Gateway

The seam every downstream product integrates against — no other Fort surface is required to consume Fort-authenticated traffic.

POST /v1/token/verify — verify a Fort bearer token; always 200 with {valid, claims|error}
GET /.well-known/jwks.json — key metadata (alg + kid only under HS256)
curl -X POST https://fort.example.com/v1/token/verify \
  -H 'Content-Type: application/json' \
  -d '{"token":"eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."}'
# {"valid":true,"claims":{"sub":"550e8400-…","role":"authenticated","iat":1749081600,"exp":1749082500}}

Auth — sign-in & second factors

Public endpoints (no JWT). On success they set the three session cookies — access_token (15 min), refresh_token (7 days), csrf_token — or return a challenge.

POST /v1/auth/login — email + password; 401 invalid_credentials, 403 method_disabled, 409 account_locked (with retry_after)
POST /v1/auth/otp/send — 6-digit email OTP; purposes login | enroll-2fa | reset-password | re-verify; 3 sends / 5 min
POST /v1/auth/otp/verify — verify OTP; login → session, other purposes → action_token
POST /v1/auth/2fa/verify — consume a challenge_token + TOTP code; sets session cookies
POST /v1/auth/magic-link — send a magic-link email (always 200; requires providers.magic_link_enabled)
GET /v1/auth/authorize/{provider} — 302 to the OAuth provider (e.g. google)
POST /v1/auth/callback — OAuth code exchange; sets session cookies
POST /v1/auth/forgot — password-reset email; always 200 (no enumeration); 3 requests / 15 min
POST /v1/auth/reset — consume the reset token and set a new password

Example: password login with an OTP step

# 1 — password accepted, but this account requires OTP
curl -X POST https://fort.example.com/v1/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"alice@example.com","password":"••••••••"}'
# {"status":"challenge","challenge":"otp","challenge_token":"eyJ..."}

# 2 — the code arrives by email; verify it (challenge_token as Bearer)
curl -c cookies.txt -X POST https://fort.example.com/v1/auth/otp/verify \
  -H 'Content-Type: application/json' \
  -H "Authorization: Bearer $CHALLENGE_TOKEN" \
  -d '{"email":"alice@example.com","code":"123456","purpose":"login"}'
# {"status":"ok"}   — session cookies set

Auth — session lifecycle

POST /v1/auth/refresh — rotate the token pair from the refresh_token cookie
GET /v1/auth/session — {valid, user} for the current cookie/Bearer session
GET /v1/auth/login-methods?email=… — which methods to render for an address (always 200)
GET /v1/auth/domain-check?email=… — {allowed}: is this email domain allowed to register
POST /v1/auth/logout — end the current session, expire cookies
POST /v1/auth/logout-everywhere — revoke every refresh token for the user

Auth — TOTP 2FA management (session required)

POST /v1/auth/2fa/enroll — begin enrollment; returns otpauth_url + base32_secret (shown once); 409 if already enrolled
POST /v1/auth/2fa/verify — first verify confirms enrollment and issues recovery codes
POST /v1/auth/2fa/recovery — consume one single-use 8-character recovery code
POST /v1/auth/2fa/disable — delete verified TOTP factors, clear the requirement
# Enroll (authenticated; cookie flow shown — note the CSRF header)
curl -b cookies.txt -X POST https://fort.example.com/v1/auth/2fa/enroll \
  -H "X-CSRF-Token: $CSRF"
# {"otpauth_url":"otpauth://totp/Fort%3Aalice%40example.com?secret=…&issuer=Fort",
#  "base32_secret":"JBSWY3DPEHPK3PXP"}   — render as QR, never log

# Confirm with the first code from the authenticator app
curl -b cookies.txt -X POST https://fort.example.com/v1/auth/2fa/verify \
  -H "X-CSRF-Token: $CSRF" -H 'Content-Type: application/json' \
  -d '{"code":"654321"}'
# {"status":"ok"}
2FA needs TOTP_ENCRYPTION_KEY. Enrollment and verification require the TOTP secret-encryption key to be set on the server — see Self-hosting.

Profile (session required)

GET /v1/profile — full profile: bilingual name_en/name_ar, role, 2FA/PIN state, active sessions, login methods
PATCH /v1/profile — partial update of profile fields
PUT /v1/profile/pin — set or change the numeric PIN (used by the lock screen)
POST /v1/profile/email/re-verify — start a two-step email change
POST /v1/profile/email/confirm — confirm the change with OTP/token
GET /v1/profile/activity — the user's recent auth activity

Admin — users (admin role required)

GET /v1/admin/users — list with search, status, method, limit, offset
GET /v1/admin/users/{id} — full user profile
PATCH /v1/admin/users/{id}/login-methods — toggle per-user method flags (password, Google, magic link, OTP/2FA required)
POST /v1/admin/users/{id}/unlock — clear a lockout
POST /v1/admin/users/{id}/force-reset — force a password reset on next login
GET /v1/admin/users/{id}/audit-log — per-user audit trail
GET /v1/admin/users/{id}/roles — RBAC roles assigned to a user
PATCH /v1/admin/users/{id}/roles — replace a user's RBAC roles

Admin — RBAC & audit

GET /v1/admin/audit — global audit log; filter by user_id, event, paginate
GET /v1/admin/roles — list roles POST — create a role
GET /v1/admin/roles/{id} — role + permissions PATCH — update DELETE — delete (non-system roles only)
PATCH /v1/admin/roles/{id}/permissions — replace the role's full permission set
POST /v1/admin/roles/{id}/permissions/toggle — toggle a single permission
GET /v1/admin/permissions — all available permissions

Branding

GET /v1/branding — public; the resolved platform's branding, consumed by the SDK before login
PATCH /v1/admin/branding — partial update; publishes fort.branding.updated to NATS
# Which platform's branding you get is resolved per request —
# the SDK passes the active platform's slug explicitly:
curl 'https://fort.example.com/v1/branding?platform=acme'

See Platforms & white-label for the resolution contract and the branding fields.

Admin — settings & access control

GET /v1/admin/settings — all namespaces, or one via ?namespace=
PATCH /v1/admin/settings/{namespace} — merge keys into auth_policy | session | providers | system
GET /v1/admin/allowed-users — list POST — add an email to the allowlist
PATCH /v1/admin/allowed-users/{id} — update DELETE — remove
GET /v1/admin/domains — list domain allowlist POST — add a domain
DELETE /v1/admin/domains/{id} — remove a domain

Admin — platforms

Registers the consuming products ("platforms") this Fort deployment serves. Full contract on the Platforms page.

GET /v1/admin/platforms — list (token hashes never returned) POST — create; the API token is returned once
GET /v1/admin/platforms/{id} — detail PATCH — update DELETE — delete (blocked for default)
POST /v1/admin/platforms/{id}/rotate-token — mint a new platform token (returned once)
GET /v1/me/platforms — platforms visible to the authenticated user (platform-switcher feed)
Deprecated alias. /v1/admin/integrations (GET/POST, DELETE /{id}) is an older alias for the platforms surface and still appears in the OpenAPI spec — prefer /v1/admin/platforms.

Setup

GET /v1/setup/state — first-run state; is_first_run=true until an admin exists
POST /v1/setup/admin — create the first admin (self-gating: 403 setup_already_complete afterwards)

Health

GET /health — {"status":"ok","service":"fort"}; no auth, safe for probes

MCP servers

Fort also mounts two Model Context Protocol servers for AI-agent tooling. They call the same internal functions as the REST handlers and verify tokens inline:

POST /mcp/admin — admin-token-gated tool surface
POST /mcp/user — any valid token; tools are self-scoped to the caller

Tool lists and JSON-RPC details are documented in the repository (internal/mcpserver/); they are not part of the OpenAPI spec.

Conventions

Errors are generic by design. Failures return {"error":"<code>"} with codes like unauthorized, forbidden, invalid_credentials, method_disabled — never stack traces, SQL, or JWT internals. Enumeration-sensitive endpoints (forgot, otp/send, magic-link, login-methods) return 200 regardless of account existence.
Rate limits. OTP send: 3 per 5 minutes per user per purpose. Forgot-password: 3 per 15 minutes per user. Login lockout threshold and duration come from the auth_policy settings namespace; a locked account answers 409 with retry_after.
