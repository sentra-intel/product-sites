Auth & roles

Motor has one primary credential — a Fort JWT — plus a Coder session fallback for operators, DSN keys for the public ingest surface, and an admin role gate in front of everything that mutates. Each surface carries exactly the auth it needs, nothing more.

Fort JWT (primary)

Fort is the platform's auth gateway. You sign in against Fort, receive a JWT, and send it to Motor on every request:

POST https://<your-fort-host>/v1/auth/login — email + password → JWT
curl -X POST https://fort.example.com/v1/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"operator@example.com","password":"••••••••"}'
# → { "token": "eyJhbGciOiJIUzI1NiIs…" }

curl -H 'Authorization: Bearer eyJhbGciOiJIUzI1NiIs…' \
  https://motor.example.com/api/me
# → your session identity (source: "fort") — the same call the dashboard makes

Motor verifies the token one of two ways, chosen by configuration:

  • Local HS256 — when FORT_JWT_SECRET is set, Motor verifies the signature locally with the shared secret. No HTTP round-trip per request.
  • Gateway verify — otherwise Motor calls Fort's POST /v1/token/verify to validate the token.
Neither configured? If both FORT_URL and FORT_JWT_SECRET are unset, Motor degrades to Coder-only auth — it never hard-fails on missing Fort config.

Coder session (secondary)

Operators who live in the shared Coder deployment can sign in with their existing session instead. Motor accepts the coder_session_token cookie and verifies it against CODER_URL/api/v2/users/me. Auth precedence is fixed: a Fort Authorization: Bearer header wins; the Coder cookie is the fallback.

curl -H 'Cookie: coder_session_token=<token>' https://motor.example.com/api/me
# → session identity (source: "coder")

Roles: admin gates the mutations

Read access needs any valid identity. Anything that changes state is gated on the Fort admin role in the JWT claims:

SurfaceRequires
GET /api/* readsAny valid Fort JWT or Coder session
Mutating REST calls (workspace/container actions, purges, CI dispatch, plugin toggles)Admin-gated; the dashboard adds an explicit confirm dialog on top
/mcp/userAny valid Fort token — the 23 read-only tools
/mcp/adminFort JWT with role=admin — read tools + the 10 mutating tools; a non-admin token gets 403

DSN auth: the public ingest surface

The Sentry-compatible ingest routes (/api/{project_id}/store/, /envelope/, /logs/, /monitors/{name}/checkin) are deliberately public — they authenticate with the project's DSN public key, not an operator session, so any product in your fleet can report without holding a Fort identity:

X-Sentry-Auth: Sentry sentry_version=7, sentry_key=<publicKey>
# also accepted: Authorization header, or a ?sentry_key= query param

The key resolves the project server-side; the project_id comes from the path. A DSN key can only submit events to its own project — it grants no reads and no actions. Full details on the ingest page.

The Feedback SDK surface (/api/issues/v1/*) works the same way with a project publishable key sent as Authorization: Bearer <publishableKey> — a public-by-design credential. Reporter identity is scoped by a signed, stateless reporter-session token (motor_rps_…) sent as X-Reporter-Session.

Optional: feedback HMAC secret

MOTOR_FEEDBACK_SECRET is an optional HMAC secret for the Feedback SDK surface — set it to sign the reporter-session flow in production deployments. See Self-hosting for where it lives in the environment.

Middleware order (what is exempt and why)

Motor's HTTP chain is CORS → auth → handler. Two groups of routes are exempt from the operator-session middleware because they carry their own auth:

corsMiddleware(cfg)( withAuth(mux) )

withAuth enforces Fort/Coder on /api/* — except:
  • the DSN-authed ingest routes   (X-Sentry-Auth / publishable key)
  • the MCP paths /mcp/admin, /mcp/user   (their own Fort auth, incl. role check)
Keep operator credentials server-side. Fort JWTs belong in servers, CLIs, and agents — never in a browser bundle. The only credentials designed to ship to browsers are the DSN public key and the feedback publishable key, which are submit-only by construction.
