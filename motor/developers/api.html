REST API reference

Around 45 paths in one OpenAPI 3.1 contract (api/openapi.yaml in the repository). Unless marked otherwise, every endpoint below accepts a Fort JWT (Authorization: Bearer) or a Coder session cookie — see Auth & roles. Examples use https://motor.example.com.

Two conventions to know. Destructive endpoints are confirm-gated in the dashboard — automate them with equivalent care. And any integration that isn't configured answers 503 with the missing env var named in the error, never a 404.

Infra control

Live state and operator actions across your infrastructure. GitHub and GCP endpoints run as the operator's own gh / gcloud CLI session per request — no stored cloud tokens.

Cloudflare

GET /api/cloudflare/dns — DNS records for the configured zone
POST /api/cloudflare/purge — purge everything in the zone cache (destructive)

Proxmox

GET /api/proxmox/resources — cluster resources: nodes, VMs, CTs, storage

Docker

GET /api/docker/containers — container list per host in DOCKER_HOSTS

GCP

GET /api/gcp/projects — GCP projects via the operator's gcloud session

GitHub & CI

GET /api/github/repos — the org's repos (?limit=)
GET /api/github/runs — recent workflow runs across repos
GET /api/ci/runs — alias of /api/github/runs
POST /api/ci/dispatch — trigger a workflow_dispatch
curl -X POST https://motor.example.com/api/ci/dispatch \
  -H 'Authorization: Bearer <fort-jwt>' \
  -H 'Content-Type: application/json' \
  -d '{"repo":"my-org/my-service","workflow":"deploy.yml","ref":"master"}'
# runs `gh workflow run …` as the operator's own gh session

Supabase & proxy hosts

GET /api/supabase — pg-meta introspection: tables (rows/size/RLS) + extensions. Schema metadata only, never rows
GET /api/proxy — Nginx Proxy Manager hosts: domains → upstream, SSL, enabled

Coder workspaces & team

GET /api/workspaces — Coder workspaces via the operator's session
POST /api/workspaces/{owner}/{name}/{action} — start | stop (confirm-gated in the UI)
GET /api/team — Coder platform members: username, name, email, status, roles

Observability reads

The read side of the observability hub — what the Sentry-compatible ingest writes:

GET /api/observability/projects — all tracked projects
GET /api/observability/projects/{id}/issues — issues (?status=&limit=&offset=)
GET /api/observability/issues/{id} — one issue with its recent events
GET /api/observability/logs — log records (?project= required; level, service, search, limit)
GET /api/observability/monitors — uptime/heartbeat monitors (?project= required)
curl -H 'Authorization: Bearer <fort-jwt>' \
  'https://motor.example.com/api/observability/logs?project=1&level=error&search=timeout&limit=50'

NATS monitor + control

GET /api/nats/servers — server overview (/varz per monitor endpoint)
GET /api/nats/streams — JetStream streams + consumers (/jsz)
GET /api/nats/connections — active connections (/connz)
POST /api/nats/streams/{name}/purge — drop all messages in a stream (destructive, confirm-gated)

Billing

Proxied to Fort's admin billing API (requires FORT_PLATFORM_TOKEN to be configured):

GET /api/billing/tokens — API keys
GET /api/billing/platforms — billing platforms
GET /api/billing/users — billing users
POST /api/billing/usage — push a usage event (Fort billing is push-based; there is no read-usage GET)
POST /api/billing/invoices/generate — generate an invoice

Feedback

Two distinct surfaces. The admin board is operator-gated; the SDK surface under /api/issues/v1 is authenticated with the project publishable key and scoped per reporter (it's what the motor-feedback widget calls):

GET /api/feedback/issues — every SDK-reported issue (operator board, Fort/Coder auth)
POST /api/issues/v1/session — mint an anonymous reporter session (motor_rps_…)
POST /api/issues/v1/issues — file a bug / change / question / discussion
GET /api/issues/v1/issues — this reporter's issues (X-Reporter-Session required)
GET /api/issues/v1/issues/{id} — one issue with comments + attachments
PATCH /api/issues/v1/issues/{id} — update triage status
DELETE /api/issues/v1/issues/{id} — delete one of the reporter's own issues
GET /api/issues/v1/issues/{id}/comments — list comments
POST /api/issues/v1/issues/{id}/comments — add a comment
POST /api/issues/v1/uploads — one attachment (multipart, max 5 MB, image/PDF/text)

Platform

GET /healthz — liveness probe, no auth
GET /api/me — the authenticated identity (source: fort or coder)
GET /api/plugins — enabled + nav-visible plugins (the dashboard sidebar source)
GET /api/plugins/all — every plugin incl. disabled (admin registry)
GET /api/plugins/{slug} — a plugin's page config: header + enabled features (a disabled plugin returns 404)
PATCH /api/plugins/{slug} — toggle enabled / nav_visible / sort_order (admin)
curl -H 'Authorization: Bearer <fort-jwt>' https://motor.example.com/api/plugins
# [ { "slug":"errors", "name_en":"Errors", "group_key":"observability",
#     "route":"/errors", "icon":"Bug", "sort_order":1 }, … ]

Errors and status codes

CodeMeaning
400Bad request — invalid input or missing configuration for the call
401No valid Fort JWT / Coder session (or invalid DSN key on ingest routes)
403Valid token, missing admin role (admin-gated surfaces)
404Resource not found — including disabled plugins and non-owned feedback issues
502Upstream API error (Cloudflare, Proxmox, Docker Engine, Coder, …)
503Integration not configured or unreachable — the error names the env var to set
