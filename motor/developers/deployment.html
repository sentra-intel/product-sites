Self-hosting & integrations

Motor is self-hosted and standalone: one Go binary, its own Postgres, its own NATS, and a Next.js dashboard. Start with zero integrations configured and light surfaces up one env var at a time — every unconfigured integration degrades gracefully instead of failing.

The binary

Everything ships in a single Cobra binary:

SubcommandWhat it does
motor serveRun the HTTP service on :8215 — REST API, ingest, and MCP in one process
motor startBring up db + nats via docker compose, then serve
motor migrateApply Atlas migrations (idempotent)
motor resetDrop and re-apply the schema (development)
motor seedSeed plugins + fixtures
motor pluginManage the DB-driven dashboard plugins: list / enable / disable / generate / agent
 
# build and run
make build          # → bin/motor
make start          # docker compose up -d (db + nats), then go run . serve

# database lifecycle — sqlc for queries, Atlas for migrations
make migrate        # apply db/migrations
make sqlc           # regenerate the type-safe query layer
make atlas          # diff a new migration from schema.sql

Ports

PortService
:8215Go HTTP API (API_PORT, or MOTOR_HTTP_ADDR to override host:port)
:5447Postgres — ParadeDB pg17 via docker compose (motor-db)
:4247NATS with JetStream via docker compose (motor-nats)
:3215Next.js web dashboard (make web)

The compose file starts only the data layer — motor-db (ParadeDB pg17) and motor-nats (JetStream enabled). The Go API and the web app run as processes in front of them.

Health probe

GET /healthz — liveness, no auth
curl https://motor.example.com/healthz
# {"status":"ok","service":"motor","version":"0.1.0","uptime":"2m30s"}

Point your load balancer, uptime checks, or Kubernetes probes here. (And once Motor is up, let it monitor everything else's heartbeats via the check-in endpoint.)

Never add GH_TOKEN or GCP_* to the environment. The absence is the security model: GitHub and GCP calls shell out to the operator's live CLI session per request, so there is no long-lived cloud credential to leak, rotate, or audit. Don't "fix" it with a stored token.

The web dashboard keeps its own web/.env.local (Coder + session secrets) — separate from the Go service's .env. Neither file is committed.

Graceful degradation

Motor is built to boot with an empty integration block. Any surface whose credentials are missing:

  • returns 503 from its API endpoint, with the error naming the exact env var to set;
  • renders as a "not configured" card in the dashboard — never a 404, never a crash;
  • comes alive as soon as the variable is set, with no schema or code change.

That makes the onboarding path incremental by design: deploy, then unlock Cloudflare today, Proxmox next week, billing whenever Fort's platform token is minted.

Database: sqlc + Atlas

  • Schema & migrations — the schema lives in db/schema.sql; Atlas diffs it into versioned migrations (make atlas), and motor migrate applies them idempotently.
  • Queries — all SQL is generated through sqlc (make sqlc); there is no raw string SQL in the server.
  • ParadeDB pg17 — the compose database image; a stock Postgres 17 with search/analytics extensions available.

Production checklist

  • TLS in front — terminate HTTPS at your reverse proxy; ingest DSNs and Fort JWTs should never travel plaintext.
  • Set Fort auth up properly — FORT_URL (+ FORT_JWT_SECRET for local verification) and a minted FORT_PLATFORM_TOKEN if you need billing surfaces. Confirm an admin-role account exists before relying on /mcp/admin.
  • Lock CORS down — MOTOR_CORS_ORIGINS should list your dashboard origin(s) only.
  • Make MOTOR_INGEST_URL the real public host — every DSN you hand out embeds it.
  • Set MOTOR_FEEDBACK_SECRET if you deploy the feedback widget publicly, and move MOTOR_UPLOAD_DIR to durable storage.
  • Wire the probe — point liveness checks at /healthz.
  • Run migrations on deploy — motor migrate before motor serve; it is idempotent.
  • Keep the CLI-delegation rule — the deploy host's gh / gcloud / coder sessions define what infra actions Motor can take. No GH_TOKEN, no GCP_*, ever.
  • Back up the database — Motor owns its data: issues, events, logs, monitors, feedback, and the plugin registry all live in its Postgres.
