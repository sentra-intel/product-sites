Sentry-compatible ingest

Motor speaks the Sentry ingest protocol. Every product in your fleet keeps its existing Sentry SDK and swaps only the DSN — errors, logs, and cron heartbeats land in Motor instead of a per-product SaaS account. No code change, no new agent, no vendor lock.

The four endpoints

POST /api/{project_id}/store/ — classic Sentry event ingest (JSON event)
POST /api/{project_id}/envelope/ — modern Sentry envelope ingest (what current SDKs send)
POST /api/{project_id}/logs/ — structured log records, single or array
POST /api/{project_id}/monitors/{name}/checkin — cron/heartbeat check-in

These routes are public by design and exempt from operator-session auth — they authenticate with the project's DSN public key instead (see below). Events are accepted with a 200; /store/ returns the event id.

DSN auth

Every request carries the DSN public key, exactly as a Sentry SDK sends it:

X-Sentry-Auth: Sentry sentry_version=7, sentry_key=<publicKey>
# also accepted: the Authorization header, or a ?sentry_key=<publicKey> query param

The key resolves the project via Motor's project_keys; the project_id is taken from the path. A DSN key is submit-only — it can never read data back or touch infra endpoints.

DSN format

Build the DSN from your public ingest host — the value of MOTOR_INGEST_URL in your deployment (it must be the externally reachable Motor API host) — plus the project's public key and numeric id:

https://<publicKey>@motor.example.com/<project_id>

Sentry SDKs derive the ingest paths from this DSN themselves (/api/<project_id>/envelope/), which is why swapping the DSN is the entire migration. Motor's dashboard shows ready-made DSNs and connection snippets per project on its Integrations page, driven by MOTOR_INGEST_URL.

Example: point a JS Sentry SDK at Motor

import * as Sentry from "@sentry/node";   // or @sentry/browser, sentry-sdk (Python), …

Sentry.init({
  dsn: "https://<publicKey>@motor.example.com/1",   // ← the only line that changes
  environment: "production",
  release: "my-service@1.4.2",
});

// everything else — captureException, breadcrumbs, contexts — works as before
Sentry.captureException(new Error("payment webhook timed out"));

Example: raw envelope POST

No SDK at all? The envelope format is newline-delimited JSON — header line, item header, item payload:

curl -X POST 'https://motor.example.com/api/1/envelope/' \
  -H 'X-Sentry-Auth: Sentry sentry_version=7, sentry_key=<publicKey>' \
  -H 'Content-Type: application/x-sentry-envelope' \
  --data-binary $'{"event_id":"9ec79c33ec9942ab8353589fcb2e04dc"}\n{"type":"event"}\n{"message":"db connection refused","level":"error","platform":"other","server_name":"worker-2"}'
# event items are ingested like /store/; non-event items are ignored

Structured logs

/logs/ accepts a single record or an array; each record carries a level, message, attributes, and optional trace/span/service fields. Query them back over GET /api/observability/logs or the dashboard's /logs surface with level/service/full-text filters.

curl -X POST 'https://motor.example.com/api/1/logs/' \
  -H 'X-Sentry-Auth: Sentry sentry_version=7, sentry_key=<publicKey>' \
  -H 'Content-Type: application/json' \
  -d '[{"level":"error","message":"retry budget exhausted","service":"scheduler",
       "attributes":{"job":"nightly-dump","attempts":5}}]'

Cron & heartbeat monitors

Monitors are auto-provisioned on first check-in — no setup step. Name the monitor in the URL and report a status of ok, error, or in_progress:

# at the end of a cron job — one line proves the job ran
curl -X POST \
  'https://motor.example.com/api/1/monitors/nightly-backup/checkin?sentry_key=<publicKey>' \
  -H 'Content-Type: application/json' \
  -d '{"status":"ok"}'

Check-ins and status history show on the dashboard's /monitors surface and via GET /api/observability/monitors?project=<id>.

How events become issues

  • Fingerprint grouping — each incoming event is fingerprinted server-side; events with the same fingerprint upsert one issue, so a thousand repeats of the same exception stay one row with a rising event count.
  • Server-side titling — the issue title is derived from the event, so heterogeneous SDKs produce consistent issue lists.
  • Read it back — issues, events, logs, and monitors are all queryable over the observability read endpoints and the MCP tools (motor_list_issues, motor_get_issue, …).

Best practices

One project per product. Give each service its own project and DSN key. Filtering, grouping, monitors, and key revocation all happen at project granularity.
Heartbeat everything scheduled. The check-in endpoint costs one curl line at the end of any cron job — and turns "the nightly job silently stopped three weeks ago" into a visible monitor state.
The DSN key is public, treat it accordingly. It ships in browser bundles by design and only grants event submission — but anyone holding it can write noise into that project. Rotate a project's key if a DSN leaks somewhere noisy.
