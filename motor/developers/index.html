Motor for developers

Motor is a self-hosted operator control plane: one authenticated API that inspects and acts on the infrastructure your platform runs on, and one observability hub that receives errors, logs, and heartbeats from every product in your fleet. Everything the dashboard shows — and every action it takes — is available to you over the same surfaces.

How the pieces connect

Your services report in over Sentry-compatible ingest; Motor stores, groups, and monitors; operators, agents, and automations read and act back out:

Into Motor
  • SDKExisting Sentry SDKs — errors via /store/ & /envelope/
  • POSTStructured logs & cron heartbeats (DSN-authed)
  • JSHuman bug reports via the Feedback widget
Motor ingests & controls
  • Events grouped into issues by fingerprint; monitors auto-provisioned
  • Live infra state: Cloudflare, Proxmox, Docker, GitHub CI, GCP, Coder, NATS
  • Actions run as the operator's own CLI session — no stored cloud tokens
Out of Motor
  • WEBThe operator dashboard — errors, logs, monitors, infra
  • MCPAI agents driving 33 tools with role gates
  • RESTYour automations, alerts & embedded panels

What you can build

🛰️

Fleet error tracking — no code change

Point the Sentry SDKs you already run at Motor's Sentry-compatible ingest. Swap the DSN, keep the SDK: events group into issues by fingerprint, logs are searchable, cron monitors auto-provision on first check-in.

🤖

AI agents that operate infra

Connect any MCP client to Motor's MCP server and an agent can triage errors, inspect JetStream, dispatch CI, or restart a container — through the same role gates as a human operator.

⚙️

Custom automations on the REST API

Every dashboard surface is a REST endpoint: script incident runbooks, feed status boards, wire alert routing — with a Fort JWT and plain HTTP.

🧩

Embedded feedback & logs panels

Drop the motor-feedback widget into any web app for in-page bug reports, and embed Motor's logs/errors/feedback panels in your own admin with @sentra/motor-admin.

One binary, every surface

Motor ships as a single Go binary that serves the REST API, the Sentry-compatible ingest, and the MCP server from the same process:

GET /api/* — infra control + observability reads (REST API)
POST /api/{project_id}/envelope/ — DSN-authed Sentry-compatible ingest (Ingest)
POST /mcp/admin · /mcp/user — Streamable-HTTP MCP, Fort-gated (MCP)
Base URLs. Local development: http://localhost:8215 · Production: your own deployment host (Motor is self-hosted) — examples across this portal use https://motor.example.com. Authenticated requests carry a Fort JWT — see Auth & roles.

5-minute quickstart

1. Run Motor

From a clone of the repository, with Docker and Go installed:

# Bring up Postgres (:5447) + NATS JetStream (:4247), then serve the API on :8215
make start

# — or step by step —
docker compose up -d          # motor-db (ParadeDB pg17) + motor-nats
go run . migrate              # apply Atlas migrations (idempotent)
go run . seed                 # plugins + fixtures
go run . serve                # HTTP API on :8215

Confirm it is up — no auth required:

curl http://localhost:8215/healthz
# {"status":"ok","service":"motor","version":"0.1.0","uptime":"2m30s"}

2. Point a Sentry SDK at Motor

Take any service already instrumented with a Sentry SDK and swap only the DSN — the host is your Motor ingest URL, the path is the Motor project id:

Sentry.init({
  dsn: "https://<publicKey>@motor.example.com/1",  // was: …@o0.ingest.sentry.io/…
});

3. Throw an error, see the issue

Trigger any exception in that service, then watch it appear — grouped by fingerprint — on the dashboard's /errors surface, or read it back over the API:

curl -H 'Authorization: Bearer <fort-jwt>' \
  https://motor.example.com/api/observability/projects/1/issues?status=unresolved

That is the whole integration: no agent to install, no SDK to replace, no vendor account.

Architecture at a glance

Your services            Sentry SDKs · motor-obs · Feedback widget
      │
      ▼
 Ingest (DSN-authed)     POST /api/{project}/store/ · /envelope/ · /logs/ · /monitors/…/checkin
      │
      ▼
 Motor core (Go :8215)   own Postgres (:5447) + NATS JetStream (:4247) — standalone by design
      │
      ├─ REST /api/*      observability reads + infra control (Fort JWT / Coder session)
      ├─ MCP /mcp/*       33 motor_* tools, Streamable-HTTP, role-gated
      └─ Web (:3215)      the DB-driven operator dashboard
  • Single binary — motor serve hosts the REST API, the ingest, and both MCP endpoints in one process.
  • CLI delegation — GitHub and GCP calls shell out to the operator's own gh / gcloud session per request. No long-lived cloud tokens are stored anywhere.
  • Standalone — Motor owns its database and its NATS. Cross-product data arrives only over public seams like the ingest endpoints.

Best practices

Keep DSNs and Fort JWTs apart. A DSN public key only lets a client submit events to one project — it can never read data or act on infra. Reads and actions always require a Fort JWT (or Coder session). Don't try to make one credential do both jobs.
Expect 503 for unconfigured integrations. A surface whose credentials aren't set returns 503 with an error naming the missing env var — not a 404, not a crash. Treat 503 as "not configured here", and handle it gracefully in your clients too.
Confirm before destructive calls. Motor is read-mostly. Endpoints that stop containers, purge streams or caches, or dispatch CI are the exception — the dashboard puts an explicit confirm in front of every one, and your automations should do the equivalent (dry-run flags, approval steps).
Give agents the least role that works. /mcp/user exposes only the 23 read tools and accepts any valid Fort token. Reserve /mcp/admin — and its 10 mutating tools — for agents that genuinely need to act.
