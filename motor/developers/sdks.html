SDKs

Three SDKs ship today, each in its own repository under the sentra-intel GitHub org so they version independently of the Motor server. And remember the zeroth SDK: any existing Sentry SDK already works against Motor's compatible ingest with a DSN swap.

motor-obsmotorobs (Go) · @sentra/motor-obs (TS)Ship logs, errors, and monitor heartbeats from your own code

motor-obs — instrument any product

motor-obs is the first-party instrumentation SDK, for code where you want direct control rather than a full Sentry SDK. It ships logs, errors, and monitor heartbeats to Motor's ingest endpoints, in Go (motorobs) and TypeScript (@sentra/motor-obs). Repo: github.com/sentra-intel/motor-obs-sdk.

# Go
go get github.com/sentra-intel/motor-obs-sdk

# TypeScript / Node
npm install @sentra/motor-obs

Both clients are configured with your Motor ingest URL, project id, and DSN public key, and speak the same wire protocol documented on the ingest page:

POST /api/{project_id}/store/ · /envelope/ — errors, grouped into issues by fingerprint
POST /api/{project_id}/logs/ — structured log records (level, message, attributes, service)
POST /api/{project_id}/monitors/{name}/checkin — cron/heartbeat check-ins (ok | error | in_progress)
Full API surface: constructor options, batching behavior, and per-language idioms are documented in the motor-obs-sdk repository README — see repository docs. Anything the SDK sends can also be sent with plain HTTP, so the SDK is a convenience, never a requirement.

motor-feedback — the in-page report widget

A drop-in browser widget that lets users file bugs, change requests, and questions from inside your app — pinned to the element and route they were looking at, with screenshots and attachments. Reports land on Motor's /api/issues/v1 surface and appear on the operator's feedback board (and via motor_list_feedback over MCP). Repo: github.com/sentra-intel/motor-feedback-sdk.

Use the npm package (@sentra/motor-feedback) in bundled apps, or the prebuilt browser bundle that Motor's web app serves as motor-feedback.js:

<script src="https://motor.example.com/motor-feedback.js"></script>
<button id="report-bug">Report a bug</button>

<script>
  // the bundle exposes a MotorFeedback global
  MotorFeedback.mountReporter({
    trigger: "#report-bug",                 // element or selector that opens the reporter
    apiBase: "https://motor.example.com/api/issues/v1",
    publishableKey: "<publishableKey>",     // the project's public feedback key
    defaultType: "bug",                      // bug | change | question | discussion
    theme: "auto",                           // or "light" / "dark"; accentColor: "#14B8A6"
  });
</script>
  • Reporter sessions — the widget identifies reporters by email and holds a signed, stateless session token (motor_rps_…), so "my issues" is scoped server-side.
  • Anchored reports — issues carry the route and an element anchor, so operators see where the report was filed.
  • Attachments — screenshots and files upload via /api/issues/v1/uploads (max 5 MB each).
  • More entry points — the bundle also exports mountFeedbackHub, mountNavigator, and openIssueView for building a fuller in-app issues experience — see repository docs.

motor-admin — Motor panels in your admin

motor-admin embeds Motor's logs, errors, and feedback panels directly into another product's admin area — a backend client plus prebuilt React panels (@sentra/motor-admin), so your support team triages reports without leaving your product. Repo: github.com/sentra-intel/motor-admin-sdk.

npm install @sentra/motor-admin
Panel props and the backend-client setup are documented in the motor-admin-sdk repository — see repository docs. The panels read from the same REST endpoints documented here, proxied through your backend so operator credentials never reach the browser.

Credential hygiene. The feedback publishable key and DSN keys are the only Motor credentials meant for browsers. Fort JWTs and FORT_PLATFORM_TOKEN stay server-side — motor-admin's backend client exists precisely so panels never hold them client-side.
