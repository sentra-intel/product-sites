Inbound webhooks

External sources push content into Sentra's enrichment pipeline through a signed inbound endpoint. Requests are authenticated per plugin with HMAC-SHA256 — no JWT involved — and every accepted envelope flows into the same capability pipeline as any other source.

Feature gate. Inbound webhook handling is controlled by the WEBHOOKS_ENABLED environment flag (default true). See Deployment for the full flag table.

The endpoint

POST /pipeline/ingest — HMAC-authenticated envelope ingest

Two headers are required on every request:

HeaderContent
X-Sentra-PluginThe plugin ID whose secret key signed the request.
X-Sentra-SignatureThe signature triple: t=<unix-seconds>,n=<nonce>,s=<hex-hmac>

Responses:

StatusMeaning
202Envelope accepted and forwarded to the pipeline.
401Invalid HMAC signature. Deliberately generic — the response never reveals which check failed (missing header, malformed signature, timestamp skew, unknown plugin, or bad MAC).
409Replay detected — the nonce was already used.
503Ingest publisher unavailable (degraded mode); safe to retry later.

Signing scheme

The signature is HMAC-SHA256 (the only accepted algorithm) over the canonical string:

<timestamp>.<nonce>.<raw-request-body-bytes>

keyed with the per-plugin secret, hex-encoded lowercase. Rules enforced by the verifier:

  • Timestamp window — t is Unix seconds; requests with |now − t| > 300 s are rejected.
  • Nonce — an opaque value up to 128 characters, unique per request. Reuse returns 409.
  • Exact bytes — the MAC is computed over the exact body bytes sent. Sign the serialised body you transmit; do not re-serialise afterwards.
  • Strict header format — exactly the keys t, n, s; duplicates or unknown keys are rejected. Signature comparison is constant-time.

Signature example (bash + openssl)

BODY='{"envelope_id":"7f2c…","source_id":"my-plugin","region":"mena","content_en":"…","ingested_at":"2026-07-19T09:00:00Z","content_hash":"…"}'
T=$(date +%s)
N=$(openssl rand -hex 16)
S=$(printf '%s.%s.%s' "$T" "$N" "$BODY" \
    | openssl dgst -sha256 -hmac "$PLUGIN_SECRET" -hex | awk '{print $NF}')

curl -X POST https://sentra.example.com/pipeline/ingest \
  -H 'Content-Type: application/json' \
  -H "X-Sentra-Plugin: my-plugin" \
  -H "X-Sentra-Signature: t=$T,n=$N,s=$S" \
  -d "$BODY"
# 202 Accepted

Verification, mirrored in Node.js

import { createHmac, timingSafeEqual } from "node:crypto";

function sign(t, nonce, body, key) {
  return createHmac("sha256", key)
    .update(`${t}.${nonce}.`).update(body)   // "<t>.<n>.<body>"
    .digest("hex");
}

// Producing a header:
const t = Math.floor(Date.now() / 1000);
const nonce = crypto.randomUUID().replace(/-/g, "");
const header = `t=${t},n=${nonce},s=${sign(t, nonce, bodyBytes, secret)}`;

Payload: the envelope

The body must be a valid Sentra envelope — the platform's content unit (contract: pkg/contracts/envelope in the repository). Core fields:

FieldNotes
envelope_idUnique ID for this envelope.
source_idOriginating source. The server overwrites the source binding with the authenticated plugin ID — a plugin can never impersonate another source.
regionRegion tag for the content.
urlCanonical URL of the item (optional).
content_en / content_arContent text — English and/or Arabic; lang declares the source language.
published_at / ingested_atTimestamps (RFC 3339). ingested_at is required.
content_hashHash of the content, used for dedup.
capabilitiesBitmask selecting which capabilities should process this envelope (see Capability SDK for the bit assignments).
scope_id / tenant_idOptional routing to a scope / tenant.
raw_payload / metadataOriginal source payload and source metadata (title, description, author, media URLs, …), preserved for provenance.

A malformed envelope that passes signature verification is rejected with 400 bad_envelope. Enriched-side fields (sentiment, topics, entities, dedup hash) are filled by the pipeline — you don't need to send them.

Retries and idempotency

Fresh signature per attempt. Nonces are single-use and timestamps expire after 5 minutes — never replay a previous request verbatim. On retry, generate a new nonce and timestamp and re-sign the same body.
Retry on 503, not on 401/409. 503 means the ingest publisher is temporarily unavailable — back off and retry. 401 is a credential/signature problem, and 409 means that exact nonce was already accepted; retrying either without fixing the cause only burns requests.
Deduplication. Envelopes carry a content_hash; the pipeline uses content hashing for dedup downstream, so re-sending the same content with a fresh signature does not produce duplicate intelligence artifacts.
Key handling. Per-plugin signing secrets are provisioned by your Sentra operator and stored server-side in a secret manager. Treat your copy like a password: server-side only, rotated if exposed.

Registering a plugin

Webhook senders are source plugins in Sentra's plugin registry. Operators manage them (create, enable/disable, connection config) through the Admin/Pipeline REST endpoints or the admin panel; each registered plugin gets its own HMAC signing key.
