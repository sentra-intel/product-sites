WebSocket: real-time updates

Sentra is real-time by architecture: capability writes fire Postgres LISTEN/NOTIFY triggers, and the WebSocket hub fans those events out to connected clients. Instead of polling, you subscribe to the scopes you care about and refetch only when something changed.

Ticket flow

WebSocket connections are authenticated with a one-time ticket, not a raw JWT in the URL:

  1. While authenticated (cookie or Bearer), request a ticket:
POST /api/ws/ticket — body optionally {"scope": "<scope-slug>"}
curl -b cookies.txt -X POST https://sentra.example.com/api/ws/ticket
# {"ticket":"…"}
  1. Connect within 10 seconds — the ticket expires after that and is single-use:
WS wss://sentra.example.com/ws?ticket=<ticket>

sessionPage in GraphQL also returns a fresh wsTicket field, so a dashboard-style client gets its first ticket for free with its session bootstrap. A legacy ?token=<jwt> parameter is still accepted on /ws but is deprecated — use tickets.

Never reuse or cache tickets. One ticket, one connection, 10-second window. Request a new ticket for every reconnect attempt.

Subscribing to scopes

After connecting, send a subscribe frame declaring the scope slugs you want. Each frame replaces the previous scope set:

{ "subscribe": ["<scope-a>", "<scope-b>"] }

Delivery rules:

  • A client with no subscription yet receives all events.
  • Events with scope "global" are delivered to everyone.
  • Otherwise an event is delivered only if its scope is in your subscription set.
  • Per-user events (notifications) are delivered only to that user's connections.

What you receive

The hub listens on Postgres NOTIFY channels wired to triggers on the intelligence tables, and forwards each event as a JSON message:

{
  "type":  "invalidate",        // "invalidate" or "push"
  "table": "narratives",        // which table changed
  "scope": "<scope-slug>",      // scope of the change ("global" = everyone)
  "id":    "<row-uuid>",        // UUID of the changed row
  "op":    "INSERT",            // INSERT | UPDATE | DELETE | UPSERT
  "user_id": "…",              // only on per-user tables (notifications)
  "payload": { /* full object — only on "push" messages */ }
}
  • invalidate — something changed; refetch the affected data (e.g. re-run the relevant GraphQL query).
  • push — the message carries the full changed object in payload; you can render it without a refetch (used e.g. for intelligence snapshot/briefing updates).

Event streams cover the live intelligence tables, including:

Complete JavaScript example

const API = "https://sentra.example.com";

async function connect(scopes, { attempt = 0 } = {}) {
  // 1. Fresh one-time ticket for every attempt (10 s TTL, single use)
  const res = await fetch(`${API}/api/ws/ticket`, {
    method: "POST",
    credentials: "include",          // send the access_token cookie
  });
  if (!res.ok) throw new Error("not authenticated");
  const { ticket } = await res.json();

  // 2. Connect immediately — don't hold the ticket
  const ws = new WebSocket(`wss://sentra.example.com/ws?ticket=${ticket}`);

  ws.onopen = () => {
    attempt = 0;
    // 3. Declare scope interest (replaces any previous set)
    ws.send(JSON.stringify({ subscribe: scopes }));
  };

  ws.onmessage = (event) => {
    const msg = JSON.parse(event.data);
    if (msg.type === "push" && msg.payload) {
      render(msg.table, msg.payload);        // full object included
    } else if (msg.type === "invalidate") {
      refetch(msg.table, msg.scope, msg.id); // e.g. invalidate a query cache
    }
  };

  // 4. Reconnect with exponential backoff + jitter, fresh ticket each time
  ws.onclose = () => {
    const delay = Math.min(30_000, 1000 * 2 ** attempt) * (0.5 + Math.random());
    setTimeout(() => connect(scopes, { attempt: attempt + 1 }), delay);
  };

  return ws;
}

connect(["<scope-a>", "<scope-b>"]);

Reconnect & client best practices

Backoff with jitter. On disconnect, retry with exponential backoff (capped, e.g. at 30 s) plus jitter so a fleet of clients doesn't reconnect in lockstep. Reset the backoff after a successful open.
Refetch on reconnect. Events that occurred while you were disconnected are not replayed. After reconnecting and resubscribing, refetch your GraphQL queries once to resync.
Subscribe narrowly. An unsubscribed connection receives every event. Send your subscribe frame right after onopen to receive only the scopes you display.
Session still matters. The ticket endpoint requires a valid session — if it starts returning 401, refresh the session (POST /api/auth/refresh) before retrying the connect loop. See Authentication.
