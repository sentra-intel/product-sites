Authentication

Every authenticated Sentra endpoint accepts a JWT — either as a Bearer token in the Authorization header or as the access_token HTTP-only cookie set at sign-in. Admin endpoints additionally require an admin role, and server-to-server callers can use a service key.

Sign-in flow

Sign in with email + password. The endpoint is rate-limited to 10 requests/minute per IP and, on success, sets access_token and refresh_token HTTP-only cookies:

POST /api/auth/sign-in
curl -c cookies.txt -X POST https://sentra.example.com/api/auth/sign-in \
  -H 'Content-Type: application/json' \
  -d '{"email":"analyst@example.com","password":"••••••••"}'
// 200 OK — session cookies set
{
  "status": "authenticated",
  "redirect_to": "/dashboard",
  "expires_at": 1789000000   // Unix timestamp
}

A wrong email or password returns 401.

Other sign-in paths exist alongside password auth — magic link (POST /api/auth/magic-link), Google OAuth (GET /api/auth/authorize/google → /api/auth/callback), and password reset (/api/auth/forgot-password, /api/auth/reset-password). GET /api/auth/login-methods?email=… reports which methods (password, magic_link, google, sso) are available for an address. See the REST reference for the full Auth tag.

Bearer header vs. cookie

Both carry the same JWT — pick the one that fits your client:

MechanismBest forHow
access_token cookie Browsers and browser-like clients Set automatically by POST /api/auth/sign-in; HTTP-only, sent on every request. Use curl -b cookies.txt to replay it.
Authorization: Bearer Scripts, backends, non-browser clients Send the JWT explicitly: Authorization: Bearer <jwt> on each request. 
# Bearer-header style
curl https://sentra.example.com/api/v1/graphql \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{"query":"{ sessionPage { session { email role } } }"}'

Refreshing and ending a session

POST /api/auth/refresh — exchanges the refresh_token cookie for a new access_token; 401 when invalid or expired
POST /api/auth/logout — invalidates the session and clears cookies (204)

Roles

The JWT carries a role claim. Two roles gate elevated surfaces:

RoleGrants
adminAdmin REST endpoints (/api/admin/*, /api/data-lab/*).
super_adminEverything admin grants, plus the superadmin GraphQL surface (superAdminPage requires isSuperAdmin in the session).

Your effective role is visible in any GraphQL response through sessionPage.session.role and sessionPage.session.isSuperAdmin. Requests to admin endpoints without a qualifying role fail with 401.

Server-to-server: X-Service-Key

Admin pipeline, prompt, and Data Lab endpoints accept a service key as an alternative to an admin JWT — designed for backend automation where no user session exists:

curl https://sentra.example.com/api/admin/pipeline/summary \
  -H "X-Service-Key: $SERVICE_KEY"

These endpoints are marked as accepting Bearer JWT (admin) or ServiceKeyAuth in the REST reference. The inbound webhook endpoint uses a different scheme again — per-plugin HMAC signatures, no JWT at all.

How tokens are verified

Sentra does not own a user database — authentication is delegated to an auth gateway. Each inbound JWT is verified by calling the gateway's token-verify endpoint (configured via FORT_URL). For local development, setting FORT_JWT_SECRET enables direct HS256 verification in-process, skipping the HTTP hop. See Deployment for the env-var contract.

Security best practices

Prefer the cookie in browsers. The access_token cookie is HTTP-only — script code can never read it, which removes the most common token-theft path. Only use Bearer headers where cookies are impractical.
Refresh proactively. expires_at in the sign-in response tells you when the access token lapses. Call POST /api/auth/refresh before that instead of retrying failed requests.
Never expose service keys or admin JWTs to clients. X-Service-Key is for trusted backends only. If a key may have leaked, rotate it and audit the admin endpoints it can reach.
Expect rate limits on auth endpoints. Sign-in is limited to 10 req/min per IP and login-methods to 3 req/min per IP. Back off on 429-style failures rather than hammering the endpoint.
