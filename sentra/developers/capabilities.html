Capability plugin SDK (Go)

A capability is a Go service that receives batches of enriched envelopes and produces intelligence artifacts — alerts, narratives, entities, briefings, and anything you design. New intelligence functions are added as capability plugins, scaffolded from one CLI command, without re-architecting the platform.

The contract

Every capability is a Go package implementing two interfaces, both satisfied by the same *Service struct:

// plugin.CapabilityPlugin — runtime contract (scheduling, dispatch)
type CapabilityPlugin interface {
    Plugin
    Capability() capability.Bitmask      // which bit this capability handles
    SubscribeTopic() string              // "sentra.capability.<id>.input.sub"
    ProcessBatch(ctx, []EnrichedEnvelope) error
    ArtifactTables() []string            // tables this capability writes to
}

// capability.Capability — admin contract (feature flags, prompts)
type Capability interface {
    ID() string
    Features() []capability.Feature
    Prompts()  []capability.PromptSlot
}

ProcessBatch is the heart of a capability: envelopes arrive, artifacts get written. Capabilities register themselves via Go init() in internal/capabilities/<slug>/runner.go, and the host loads them from the registry.

Feature flags and prompt templates

Each capability declares a static catalog of Feature flags and PromptSlot templates. At boot, these are upserted into capability_features and capability_prompts — after which admins can toggle features and edit prompts at runtime, without a redeploy. The boot upsert refreshes metadata but never overwrites admin-edited values.

capability.Feature{
    ID:             "primary-flow",
    Name:           "Daily Briefing Synthesis",
    Description:    "Generates a daily bilingual executive briefing per scope...",
    DefaultEnabled: true,
    Priority:       100,
}

capability.PromptSlot{
    Slug:            "intel-synthesis-prompt",
    DefaultTemplate: `You are a senior intelligence analyst...`,
    DefaultLanguage: "en",
    Variables:       []string{"scope", "narratives_json", "alerts_json"},
}

Two PromptSlots with the same Slug and different DefaultLanguage values declare a bilingual pair. Prompt edits through the Admin/Prompts API are versioned — a new version row is inserted, never a mutation.

Scaffolding a capability

Option A: sentra-hub capability generate

sentra-hub capability generate \
  --slug my-cap \
  --name-en "My Cap" \
  --name-ar "قدرتي" \
  --desc-en "Does X for operators" \
  --desc-ar "يقوم بـ X للمشغلين" \
  --icon sparkles \
  --color '#7C3AED'

Add --dry-run to preview every file without writing, or --ai --why "…" to draft the default prompt templates via the configured LLM. The scaffolder creates the Go service, features, and plugin files under internal/capabilities/<slug>/, the runner.go registration, a React section component for the dashboard, and the frontend route/icon/label registry entries.

Option B: agentic scaffold

sentra-hub capability agent "Analyse diplomatic positions from alerts, \
  track per-country stances, link to related entities"

This asks the configured LLM to design a PluginSpec, then drives an AI coding agent (if installed) to implement the full capability from the spec — otherwise it prints the spec and the equivalent generate command to run manually.

Walkthrough: after scaffolding

  1. Implement ProcessBatch in internal/capabilities/<slug>/internal/plugin.go — envelopes in, artifacts out. Use pgxpool with parameterised ($1) queries.
  2. Add the bitmask constant in pkg/contracts/capability/capability.go — append-only, never reuse bits: 
    MyCap Bitmask = 1 << 18  // append only — never reuse bits
    Update All() and the TopicName() switch in the same file.
  3. Wire the subscriber in the serve command — add an import + registration line at the marked SENTRA-CAPABILITY-IMPORTS-START block.
  4. Enable it in the superadmin panel at /superadmin/capabilities/<slug> — confirm the Feature flags upserted, then toggle the capability on for the relevant scopes.
  5. Write tests — a unit test for the core logic, a ProcessBatch happy path, and a bilingual parity test (below).

Compile-time checklist:

var _ plugin.CapabilityPlugin = (*Service)(nil)  // compile-time assertion

func (s *Service) ID() string                      { return "<slug>" }
func (s *Service) Capability() capability.Bitmask  { return capability.MyCap }
func (s *Service) SubscribeTopic() string          { return "sentra.capability.<slug>.input.sub" }
func (s *Service) ArtifactTables() []string        { return []string{"public.<slug>_items"} }

Runtime dispatch

Envelopes reach capabilities via the Postgres NOTIFY channel envelope_ingested: the host reads each envelope's capability bitmask and dispatches it to every matching subscriber. In local dev all subscribers run as goroutines inside the bridge process; in production each capability can run as its own Deployment. Capabilities may also define their own triggers — the briefing capability, for example, fires synthesis when a content cluster forms (≥3 envelopes from ≥2 distinct sources within a rolling 4-hour window), falls back to an envelope-count accumulator, and always runs once daily at 06:00 UTC.

Bilingual output requirement

Every artifact must carry both languages. All user-facing text fields are EN/AR pairs (title_en/title_ar, …). When the source is Arabic-only, translate to fill the English field; when English-only, produce the Arabic via AI. This is enforced by a shared test pattern: 
func TestBilingualOutput(t *testing.T) {
    for _, artifact := range artifacts {
        require.NotEmpty(t, artifact.TitleEN, "title_en must be set")
        require.NotEmpty(t, artifact.TitleAR, "title_ar must be set")
    }
}

Best practices

Route AI calls through the inference gateway. All LLM calls go through the platform's AI gate (which targets the configured OpenAI-compatible endpoint or falls back to direct provider keys) — never call provider SDKs directly from a capability.
Design prompts for admin editing. Everything an operator might tune belongs in a PromptSlot or Feature, not a hardcoded constant — that's what makes runtime tuning without redeploys possible.
Declare your artifact tables. ArtifactTables() is how the platform knows what you write. Keep it accurate; it drives observability and admin tooling.
