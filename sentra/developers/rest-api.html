REST API reference

The Sentra Bridge REST API (OpenAPI 3.1.0) covers auth, health, WebSocket tickets, the GraphQL transport, the inbound pipeline, and the admin surface. The full machine-readable contract lives at api/openapi.yaml in the repository.

Base URLs

EnvironmentBase URL
http://localhost:8090Local development (default PORT)
https://sentra.example.comYour deployment host (self-hosted)

Conventions

  • Content type — request and response bodies are application/json.
  • Authentication — unless marked public, endpoints require a JWT (Bearer header or access_token cookie). Endpoints labeled admin/service-key require role=admin/super_admin in the JWT claims, or an X-Service-Key header. See Authentication.
  • Errors — failures return an error object; details are intentionally generic:
{ "error": "…", "message": "…" }
  • Rate limits — documented per endpoint where they exist (sign-in: 10 req/min per IP; login-methods: 3 req/min per IP).
  • Pagination — list-style reads are primarily served by GraphQL, which uses limit/offset arguments. The REST admin list endpoints return full collections.

Health

GET /health — liveness probe; no auth
curl https://sentra.example.com/health
{
  "status": "ok",
  "service": "sentra-bridge",
  "db": "connected",
  "version": "0.1.0"
}

Public

No authentication required.

GET /api/branding — organisation branding for the login page (orgName, logoUrl, primaryColor)
GET /api/setup/state — first-run detection ({"is_first_run": bool})
GET /api/setup/status — setup completion status ({"completed": bool})
GET /api/platform/flags — boolean feature flags safe for unauthenticated callers (e.g. demo_mode)
POST /api/rum — Real User Monitoring ingest (LCP/INP/CLS telemetry + JS errors); returns 204

Auth

POST /api/auth/sign-in — email + password sign-in; sets session cookies; rate-limited 10 req/min per IP
POST /api/auth/magic-link — send a magic-link email
GET /api/auth/authorize/google — initiate Google OAuth redirect (302)
GET /api/auth/callback — OAuth callback; redirects to the dashboard after token exchange
POST /api/auth/callback — OAuth callback (POST variant)
POST /api/auth/refresh — refresh the session using the refresh_token cookie
POST /api/auth/logout — invalidate the session and clear cookies (204)
POST /api/auth/forgot-password — send a password-reset email
POST /api/auth/reset-password — exchange an OTP token and set a new password
GET /api/auth/login-methods?email=… — available auth methods for an address; rate-limited 3 req/min per IP
curl -c cookies.txt -X POST https://sentra.example.com/api/auth/sign-in \
  -H 'Content-Type: application/json' \
  -d '{"email":"analyst@example.com","password":"••••••••"}'
# 200 → {"status":"authenticated","redirect_to":"/dashboard","expires_at":1789000000}
# 401 → invalid credentials

WebSocket

POST /api/ws/ticket — issue a one-time WebSocket auth ticket (10 s TTL); requires auth
GET /ws?ticket=… — WebSocket upgrade (101); authenticated by the one-time ticket
curl -b cookies.txt -X POST https://sentra.example.com/api/ws/ticket
# {"ticket":"…one-time token, valid 10 seconds…"}

The ticket request body optionally carries a scope: {"scope": "<scope-slug>"}. A legacy ?token=<jwt> query parameter on /ws is still accepted but deprecated. Full flow, channels, and a client example: WebSocket guide.

GraphQL

POST /api/v1/graphql — the single GraphQL endpoint; body: {query, variables, operationName}

GraphQL errors are returned in the response body (errors array), not as HTTP status codes. Schema and examples: GraphQL guide.

Pipeline

POST /pipeline/ingest — plugin envelope ingest; HMAC-authenticated via X-Sentra-Plugin + X-Sentra-Signature (no JWT)

Responses: 202 accepted · 401 invalid HMAC signature · 409 replay detected (nonce reuse) · 503 ingest publisher unavailable. Signing scheme and examples: Webhooks guide.

Admin / Pipeline

All endpoints in this tag require an admin JWT or X-Service-Key.

GET /api/admin/pipeline/summary — pipeline health summary
GET /api/admin/pipeline/plugins — list all source + capability plugins
GET /api/admin/pipeline/plugins/{id} — get a plugin by ID or slug
PATCH /api/admin/pipeline/plugins/{id} — update plugin fields
DELETE /api/admin/pipeline/plugins/{id} — delete a plugin (204)
POST /api/admin/pipeline/plugins/{id}/enabled — toggle plugin enabled state
GET /api/admin/pipeline/plugins/{id}/connection — get core-plugin connection config (cortex-core, scout-core, fort-core)
PUT /api/admin/pipeline/plugins/{id}/connection — update core-plugin connection config (base_url, api_key, nats_url, token, platform_token)
GET /api/admin/pipeline/capabilities — list all capability plugins
GET /api/admin/pipeline/capabilities/{id} — get a capability by ID or slug
PATCH /api/admin/pipeline/capabilities/{id} — update capability fields
PATCH /api/admin/pipeline/capabilities/{id}/enabled — toggle capability enabled state
GET /api/admin/pipeline/capabilities/{id}/workflow — get capability workflow steps + 7-day metrics
PUT /api/admin/pipeline/capabilities/{id}/workflow — atomic replace of capability workflow steps
GET /api/admin/pipeline/scopes — list scopes visible in pipeline admin
GET /api/admin/pipeline/sources — list all source configurations
curl https://sentra.example.com/api/admin/pipeline/summary \
  -H "X-Service-Key: $SERVICE_KEY"

Admin / Prompts

Admin JWT or X-Service-Key required.

GET /api/admin/prompt-templates?capability=… — list prompt templates, optionally filtered by capability
GET /api/admin/prompt-templates/{id} — get a single template with its full body
PATCH /api/admin/prompt-templates/{id} — versioned write: inserts a new version row, never mutates the existing one
Prompt edits are versioned. A PATCH always creates a new version; previous versions are preserved. This is how capability prompts are tuned at runtime without a redeploy — see the Capability SDK.

Admin / Data Lab

Admin JWT or X-Service-Key required.

GET /api/data-lab/plugins — list plugins with pipeline observability data
GET /api/data-lab/items — list pipeline processed items
GET /api/admin/edge-migration/status — edge function migration tracker
