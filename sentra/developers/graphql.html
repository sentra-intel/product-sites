GraphQL API

GraphQL is Sentra's primary read API — the same API the dashboard uses. Page-level queries return everything a screen needs in one round-trip: intelligence feeds, briefings, session data, and capability configuration.

Endpoint and authentication

POST /api/v1/graphql

Requests are standard GraphQL-over-HTTP JSON bodies — {"query": "...", "variables": {...}, "operationName": "..."}. Authenticate exactly like any other endpoint: Authorization: Bearer <jwt> or the access_token cookie from sign-in (see Authentication). All queries require authentication.

Errors come back in the response body's errors array with HTTP 200 — check for it before reading data.

curl -b cookies.txt -X POST https://sentra.example.com/api/v1/graphql \
  -H 'Content-Type: application/json' \
  -d '{"query":"{ sessionPage { session { email role } } }"}'

Schema overview

The query surface is organised around page queries — one query per screen — plus intelligence bundles and CRM-style lists:

QueryWhat it returns
sessionPagePer-session globals, fired once per session: session, branding, capability nav, admin settings, lenses, and a fresh wsTicket.
dashboardPage(scopeSlug, limit)Full dashboard payload for a scope: session, capability nav, intelligence (narratives/alerts/events/entities), unified feedData, and the user's interestProfile.
multiScopeIntelligence(scopeSlugs, limit)Intelligence bundle (narratives, alerts, events, entities, snapshot, briefing) per requested scope.
criticalScopeBadges(scopeSlugs, sinceIso)Count of active critical/high-urgency narratives per scope since a timestamp (clamped to a 24 h lookback).
briefingPage(scopeSlug)Scope briefing payload — brief text plus attached alerts and entities.
personalBriefingPageThe calling user's personal briefing.
briefingsListPage(limit, offset)Paginated unified briefings list (personal + scope rows).
capabilityPage(slug)DB-driven config for one capability page: feature flags, layout, prompts. Null when the slug is unknown.
superAdminPageSuperadmin panel bootstrap payload. Requires isSuperAdmin.
clients / clientPage / clientConfigCRM-style client registry: paginated list, full client detail in one round-trip, and the create-client wizard config.
researches / researchPage / researchConfigResearch Center: paginated research list, full workspace detail, and the deliverable-type registry.
Read-only by design. The GraphQL schema exposes queries only — writes go through the REST API (for example, research writes go via /api/research per the schema docs).

Example: alerts and narratives feed for a scope

query Dashboard($scope: String!) {
  dashboardPage(scopeSlug: $scope, limit: 10) {
    intelligence {
      alerts    { id slug severity titleEn titleAr createdAt }
      narratives { id slug urgency status titleEn titleAr createdAt }
    }
  }
}

Example: multi-scope intelligence with snapshots

query MultiScope($slugs: [String!]!) {
  multiScopeIntelligence(scopeSlugs: $slugs, limit: 5) {
    scopeSlug
    alerts   { id severity titleEn }
    events   { id eventType titleEn occurredAt locationEn significance }
    entities { id nameEn nameAr type country sentiment }
    snapshotJson   # raw snapshot JSONB serialised as a JSON string
    briefingJson   # latest briefing row, serialised as a JSON string
  }
}

Fields ending in Json (and the JSON scalar fields such as feedData) return serialised JSON — parse them client-side.

Example: entity data

Entities arrive inside the intelligence bundles, with bilingual names, type, country, and sentiment:

query {
  dashboardPage(scopeSlug: "<scope-slug>") {
    intelligence {
      entities { id slug nameEn nameAr type country role sentiment imageUrl projectIds }
    }
  }
}

Conventions

  • Pagination — list queries take limit and offset arguments. Defaults and caps are documented per query in the SDL (e.g. briefingsListPage: default 20, max 100; clients: default 100, max 500).
  • Filtering — list queries accept a filter search string (matched against resolved names) and, where relevant, status-style args such as researches(subjectType, status). Scope-based queries take scopeSlug / scopeSlugs.
  • Bilingual fields — user-facing text comes in pairs: titleEn/titleAr, nameEn/nameAr, descriptionEn/descriptionAr. The Arabic member may be null; fall back to English.
  • Timestamps — returned as strings (e.g. createdAt); time-window args take ISO strings (sinceIso).

Where the schema lives

The full SDL is exported to api/schema.graphql in the repository and is the authoritative, documented contract — every query above carries a doc comment there. Pair GraphQL reads with the WebSocket stream to refetch only when data actually changes.
