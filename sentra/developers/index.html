Sentra Developer Portal

Sentra turns raw content — articles, social posts, broadcast clips — into structured intelligence: alerts, narratives, events, entity networks, and executive briefings. Everything the dashboard shows is available to you over the same APIs.

What the API gives you

Connect once, and intelligence flows both ways — your content in, structured intelligence out, in real time:

Into Sentra
  • POSTSigned webhooks — your content into the pipeline
  • NATSCollection-layer envelopes (optional seam)
  • GOYour own capability plugins
Sentra enriches
  • Entities, sentiment, severity extracted — EN & AR
  • Signals clustered into narratives & events
  • Briefs synthesized, networks built — with provenance
Into your systems
  • GQLStructured reads — one query, full payload
  • WSLive push the moment artifacts land
  • RESTAuth, admin & operational automation

What you can build

📥

Intelligence consumers

Pull alerts, narratives, events, entities and briefings for a scope over GraphQL — and keep them live over WebSocket instead of polling. Route critical alerts into ops-room screens, Slack/Teams, or a SOC.

📤

Content producers

Push signed content envelopes into the enrichment pipeline through the inbound webhook endpoint — internal reports and third-party feeds become first-class intelligence.

🧩

New capabilities

Extend the platform itself with a Go capability plugin that consumes enriched envelopes and writes new artifacts — scaffolded from one CLI command.

⚙️

Operational integrations

Server-to-server automation against the admin pipeline API with a service key — feed BI warehouses, trigger downstream workflows, embed live data in internal portals.

Three API surfaces, one binary

Sentra ships as a single Go binary (sentra-hub) that serves all three surfaces from the same process:

POST /api/v1/graphql — primary read API for dashboard-style data (GraphQL)
GET /api/* — REST surface: auth, health, tickets, admin pipeline (REST API)
WS /ws?ticket=… — real-time updates via one-time auth tickets (WebSocket)
Base URLs. Local development: http://localhost:8090 · Production: your own deployment host (Sentra is self-hosted) — examples below use https://sentra.example.com. All authenticated requests carry a JWT — see Authentication.
Client SDKs are on the roadmap. There are no published client SDKs today — the REST, GraphQL, and WebSocket contracts documented here are the supported integration surface, and they are stable enough to call directly from any HTTP client. The Go capability plugin SDK (for extending the platform itself) is available now.

5-minute quickstart

1. Run Sentra

From a clone of the repository, with Docker and Go installed:

# Build the binary
make build                          # → ./bin/sentra-hub

# Full local stack: docker compose up -d (Postgres + NATS)
# + migrations + the API server on :8090
make start

# — or step by step —
docker compose up -d                # Postgres :5444, NATS :4244
./bin/sentra-hub migrate            # apply db/migrations (idempotent)
./bin/sentra-hub seed --profile dev  # seed a dev dataset
./bin/sentra-hub serve              # bridge API on PORT (default 8090)

Confirm it is up — no auth required:

curl http://localhost:8090/health
# {"status":"ok","service":"sentra-bridge","db":"connected","version":"0.1.0"}

2. Make your first authenticated request

Sign in to get session cookies (access_token + refresh_token):

curl -c cookies.txt -X POST http://localhost:8090/api/auth/sign-in \
  -H 'Content-Type: application/json' \
  -d '{"email":"you@example.com","password":"••••••••"}'
# {"status":"authenticated","redirect_to":"/dashboard","expires_at":1789000000}

3. Run your first GraphQL query

curl -b cookies.txt -X POST http://localhost:8090/api/v1/graphql \
  -H 'Content-Type: application/json' \
  -d '{"query":"{ sessionPage { session { email role isSuperAdmin } wsTicket } }"}'

The response includes your session and a wsTicket you can use immediately to open a live WebSocket connection.

Architecture overview

One binary, one PostgreSQL database, real-time by design:

Collection layer (optional, NATS)          SCOUT_NATS_URL — envelopes.raw
      │
      ▼
 scoutfeed consumer      INSERT into raw_envelopes + pg_notify(…)
      │
      ▼
 capability subscribers  alerts · narratives · events · entities ·
      │                  briefings · coverage · tasks · notifications …
      │  pg_notify on artifact writes
      ▼
 WebSocket hub           wss://…/ws?ticket=<one-time-ticket>
      │
      ▼
 Your clients            GraphQL / REST reads + live WS updates
  • Single binary — sentra-hub serve embeds the REST/GraphQL/WebSocket server, the plugin host, and the capability subscribers.
  • PostgreSQL as the backbone — every artifact is a table row; Postgres LISTEN/NOTIFY drives the WebSocket hub, so live updates need no separate message broker.
  • Optional seams, cleanly decoupled — collection arrives over NATS (SCOUT_NATS_URL), AI inference routes through an OpenAI-compatible HTTP gateway (CORTEX_HTTP_ADDR, with direct provider keys as fallback), and auth delegates to a gateway (FORT_URL). Each seam is env-configured and Sentra runs standalone without any of them.

Best practices

Read over GraphQL, listen over WebSocket. Page queries like dashboardPage return a full payload in one round-trip. Use the WebSocket stream to know when to refetch — never poll.
Treat tickets as disposable. WebSocket tickets are one-time and expire in 10 seconds. Request a fresh ticket for every connect and reconnect — never cache one.
Expect bilingual data. Every user-facing text field comes in an EN/AR pair (titleEn/titleAr, name_en/name_ar). The Arabic field can be null; fall back to English in your UI.
Keep credentials server-side. Service keys (X-Service-Key) and webhook HMAC secrets must never reach a browser. Browser clients should rely on the HTTP-only cookie session.
