Deployment & self-hosting

Sentra is a single Go binary plus its own PostgreSQL — self-hostable on Docker, Kubernetes, or a cloud container runtime, including sovereign/on-prem environments. This page covers the environment contract and the deployment paths.

Requirements

  • The sentra-hub binary — the backend image builds with CGO enabled (Tesseract/Leptonica OCR is linked for the copilot image tool); the provided Dockerfile handles this.
  • PostgreSQL 17 — Sentra owns its database (local dev runs the paradedb/paradedb:latest-pg17 image). Row Level Security is enabled on every table.
  • Optional services — NATS (envelope collection seam), Redis (config + capability nav cache), an AI inference gateway or direct provider keys, and the auth gateway.

Docker

# Backend (CGO required for Tesseract/Leptonica OCR)
docker build -f infra/docker/Dockerfile.backend -t sentra-backend .

# Web dashboard
docker build -f infra/docker/Dockerfile.web -t sentra-web .

Local stack with docker compose

The repository's docker-compose.yml provides the data services; the binary runs migrations and serves:

docker compose up -d          # Postgres :5444, NATS :4244
sentra-hub migrate            # apply db/migrations (idempotent)
sentra-hub serve              # bridge API on PORT (default 8090)

# or all three in one step:
sentra-hub start

Kubernetes

kubectl apply -f infra/k8s/deployment.yaml
kubectl -n sentra create secret generic sentra-secrets \
  --from-literal=database_url=<postgres-url> \
  --from-literal=jwt_secret=<secret> \
  --from-literal=supabase_service_key=<key> \
  --from-literal=fort_url=<url> \
  --from-literal=fort_platform_token=<token> \
  --from-literal=cortex_http_addr=<url> \
  --from-literal=cortex_token=<token> \
  --from-literal=scout_nats_url=<nats-url> \
  --from-literal=scout_token=<token>

Kubernetes manifests live under infra/k8s/ in the repository. In production, capability subscribers can run as separate Deployments rather than in-process goroutines (see runtime dispatch).

Health probe

GET /health — liveness/readiness probe; no auth
curl http://localhost:8090/health
# {"status":"ok","service":"sentra-bridge","db":"connected","version":"0.1.0"}

db: "connected" confirms the database link — wire this endpoint into your container platform's liveness and readiness checks.

Migrations

The schema lives in db/migrations/ as plain timestamped SQL files (370+), applied by the built-in runner:

sentra-hub migrate      # idempotent — skips already-applied files
  • Applied versions are recorded in the database, keyed by each filename's timestamp prefix; re-running is safe.
  • Migrations are append-only — schema changes ship as new timestamped files, never edits to applied ones.
  • DATABASE_URL (or SENTRA_DB_URL) selects the target database.
  • Run migrate against the database before starting a new binary version.

Production checklist

Lock down CORS. Set CORS_ORIGINS to your exact dashboard origin(s). Never deploy with a wildcard.
Secrets belong in a secret store. Database URL, JWT secret, service keys, gateway tokens, and webhook HMAC keys should come from your platform's secret manager (as in the Kubernetes example above), not from image layers or plain env files.
Disable what you don't use. The feature flags let you turn off the webhook handler, copilot, or admin routes per environment — a smaller surface is an easier one to defend.
Migrate, then roll. Apply migrations before rolling the new binary, and gate the rollout on GET /health returning db: "connected".
Dev-only settings stay in dev. APP_ENV=development (dev seeds) and FORT_JWT_SECRET (local token verification that skips the auth gateway) are development conveniences — don't carry them into production.
